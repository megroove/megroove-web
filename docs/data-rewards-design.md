# データ提供リワード機構 設計案・ロードマップ

Megroove のユーザーが、自分のコーヒー記録データを **自らの意思で** カフェ・ロースター・器具メーカー等のパートナー企業に提供し、対価としてポイント・クーポン等の報酬を受け取れる仕組みの設計案。

> 大前提: **本アプリのサーバーレス構造は維持する。**
> アプリ運営側はサーバーを持たず、データはユーザー端末（IndexedDB）から
> **各パートナー企業が運営するサーバーへ、提供の都度、直接送信** される。
> アプリ運営者はデータの経路にも保管にも一切関与しない。

---

## 1. コンセプトと原則

「データの民主化」— 記録データの所有者はユーザー自身であり、その価値の対価はユーザー自身が受け取る。

1. **ユーザー主権** — 提供は常にオプトイン。何を・誰に・どの粒度で送るかをユーザーが決め、送信前に実際のJSONをプレビューできる。いつでも撤回できる。
2. **サーバーレス維持** — アプリ運営者は中央サーバーを持たない。ブラウザ ⇄ パートナーサーバーの直接通信のみ。ポイント台帳もユーザー端末に置く（署名付きレシートで検証可能にする）。
3. **最小化と匿名化** — 送信データはスコープ単位で最小化。写真・自由記述メモは既定で除外。ユーザーIDはパートナーごとに異なる仮名IDを使い、企業間の突合を防ぐ。
4. **透明性** — パートナー一覧・要求スコープ・報酬内容は公開レジストリ（Git管理）で誰でも監査できる。

## 2. 全体アーキテクチャ

```
┌────────────────────────┐
│  GitHub Pages（静的ホスティング・従来どおり）      │
│  ├─ Megroove アプリ本体                        │
│  └─ パートナーレジストリ partners.json（署名付き） │
└──────────┬─────────────┘
           │ ①レジストリ取得（静的ファイルfetch）
┌──────────▼─────────────┐
│  ユーザーのブラウザ                             │
│  ├─ IndexedDB（記録データ・従来どおり）          │
│  ├─ 同意ストア（パートナー別の同意・スコープ）     │
│  ├─ 提供パッケージ生成（匿名化・スコープ抽出）     │
│  └─ ポイント台帳（受領レシートの蓄積）            │
└──────────┬─────────────┘
           │ ②HTTPS直接POST（都度接続・CORS）
┌──────────▼─────────────┐
│  パートナー企業のサーバー（各社が自前で運営）       │
│  ├─ POST /megroove/v1/provisions（データ受領）  │
│  ├─ GET  /megroove/v1/offer（報酬オファー提示）  │
│  └─ 署名付きレシート発行（Ed25519）              │
└────────────────────────┘
```

### 2.1 パートナーレジストリ（partners.json）

サーバーを持たずにパートナーを管理する要。アプリと同じ GitHub Pages に置く静的 JSON。
更新は Git の PR 経由 → 変更履歴がそのまま監査ログになる。改ざん防止のため運営鍵で署名する。

```jsonc
{
  "version": 1,
  "updatedAt": "2026-07-01",
  "signature": "<運営のEd25519署名（partners配列に対する）>",
  "partners": [
    {
      "id": "example-roastery",
      "name": "Example Roastery",
      "category": "roaster",            // cafe | roaster | manufacturer | research
      "endpoint": "https://data.example-roastery.jp/megroove/v1",
      "publicKey": "<パートナーのEd25519公開鍵（レシート検証用）>",
      "scopes": ["brew.params", "brew.rating", "bean.master"],
      "rewardSummary": "提供1回につき100pt（100円相当のクーポン）",
      "privacyPolicyUrl": "https://example-roastery.jp/privacy",
      "verifiedAt": "2026-06-15"
    }
  ]
}
```

- パートナー掲載前に運営が審査（実在確認・プライバシーポリシー・受領エンドポイントの動作確認）。
- CSP は `connect-src` にレジストリ掲載ドメインを許可する必要がある。静的CSPでは未知ドメインを列挙できないため、**「提供実行時のみ、ユーザー明示操作を起点に fetch する」前提で `connect-src https:` に緩和**するか、レジストリ更新とCSP更新を同時デプロイする運用にする（Phase 1 の確認事項）。

### 2.2 同意モデル（端末内保存）

IndexedDB に `consents` ストアを追加。パートナー×スコープ単位で管理。

```ts
interface PartnerConsent {
  partnerId: string
  scopes: DataScope[]        // 許可したスコープ
  granularity: 'record' | 'aggregated'  // 明細か集計値のみか
  grantedAt: string
  expiresAt?: string         // 期限付き同意（例: 1年）
  revokedAt?: string         // 撤回（以後の送信を停止）
}
```

- 撤回後の**過去データの削除請求**は、パートナーのプライバシー窓口経由（レジストリにURL必須）。アプリからワンタップで削除請求メール/フォームに誘導する。

### 2.3 データスコープ（提供単位）

| スコープ | 内容 | 除外されるもの |
|---|---|---|
| `brew.params` | 粉量・湯量・比率・挽き目・湯温・抽出時間・注湯回数・器具タイプ | — |
| `brew.rating` | 星評価・カッピング5軸・フレーバーチップ | メモ（自由記述） |
| `bean.master` | 豆の産地・品種・精製方法・焙煎度・焙煎からの日数 | 豆の購入メモ |
| `cafe.visits` | ドリンク種別・サイズ・評価・価格帯（丸め） | カフェ名は選択制・メモ除外 |
| `stats.monthly` | 月次集計のみ（杯数・平均評価・産地分布） | 明細すべて |

**常に除外（送信不可能）**: 写真（photoDataUrl）、自由記述メモ、カフェイン/就寝設定（健康関連）、正確なタイムスタンプ（日付単位に丸める）。

### 2.4 仮名ID（パートナー間の突合防止）

初回に端末内で `userSecret`（乱数32byte）を生成し IndexedDB に保存。
パートナーごとの仮名IDは `pseudoId = HMAC-SHA256(userSecret, partnerId)` で導出する。

- 同一パートナーには常に同じIDで継続性（累積ポイント）を担保。
- パートナーが異なればIDも異なるため、企業間でユーザーを突合できない。
- `userSecret` はエクスポートJSONに含めてバックアップ対象にする（機種変更でポイント継続）。

### 2.5 提供パッケージ（送信ペイロード）

```jsonc
{
  "format": "megroove-provision/v1",
  "pseudoId": "<パートナー別仮名ID>",
  "partnerId": "example-roastery",
  "consentScopes": ["brew.params", "brew.rating"],
  "period": { "from": "2026-06-01", "to": "2026-06-30" },
  "records": [ /* スコープ抽出・匿名化済みの記録。日付は day 精度に丸め */ ],
  "clientVersion": "1.x.x"
}
```

生成はすべてクライアント側（端末内）で完結。送信前に**この JSON そのものをプレビュー画面で表示**し、ユーザーが「この内容を送る」を押して初めて fetch する。

### 2.6 報酬とポイント台帳（サーバーレスでの成立方法）

中央のポイントバンクは持てないため、**ポイントは「パートナー別」**（店舗ポイントカードと同じモデル）とする。

1. パートナーはデータ受領時に**署名付きレシート**を返す:
   ```jsonc
   {
     "receiptId": "...",
     "pseudoId": "...",
     "dataHash": "<受領ペイロードのSHA-256>",
     "points": 100,
     "reward": { "type": "coupon", "code": "MEG-XXXX", "expiresAt": "..." },
     "issuedAt": "...",
     "signature": "<パートナー秘密鍵によるEd25519署名>"
   }
   ```
2. アプリはレジストリの公開鍵で署名検証し、端末内の `receipts` ストアに保存 → これがポイント台帳。
3. 利用時: クーポンコード/QRを店頭・ECで提示。パートナー側は自社サーバーで `pseudoId` ごとの累積ポイントを保持しているため、端末側台帳は「見える化」であり、消失しても復元可能。

### 2.7 パートナー向け標準API仕様（受入必須要件）

パートナーのオンボーディングコストを下げるため、標準仕様と参照実装ドキュメントを提供する。

```
GET  {endpoint}/offer
  → 現在の報酬オファー（スコープ別ポイント、キャンペーン）
POST {endpoint}/provisions
  Content-Type: application/json（megroove-provision/v1）
  → 201 + 署名付きレシート
必須: CORS（Access-Control-Allow-Origin にアプリのオリジン）、HTTPS、レート制限
```

## 3. 法務・コンプライアンス（日本）

実装前に必ず整理する項目。リリースのブロッカーになり得る。

| 項目 | 対応 |
|---|---|
| 個人情報保護法 | 送信データはユーザー自身の操作による本人提供であり運営は関与しないが、**アプリのプライバシーポリシーに仕組みを明記**。パートナーには掲載条件として利用目的の特定・公表を義務付ける |
| 外部送信規律（改正電気通信事業法） | ユーザー端末から外部への情報送信に該当し得るため、**送信先・送信内容・利用目的の事前通知/公表**をアプリ内に実装（提供プレビュー画面がこれを兼ねる設計にする） |
| 景品表示法 | ポイント・クーポンの表示はパートナー責任だが、レジストリ審査で表記をチェック |
| 未成年対応 | 同意フローに年齢確認の注意文言 |

## 4. ロードマップ

### Phase 0 — 土台（アプリ本体のブラッシュアップと並行）✅ 実装済み（2026-07-09、プライバシーポリシーを除く）
**ゴール: パートナーがいなくても価値がある「自分のデータの構造化」を完成させる**
- [x] データスキーマのバージョニング（`megroove-provision/v1`、schemaVersion: 1）
- [x] 匿名化エンジン実装（スコープ抽出・丸め・除外を行う純関数群 `src/provision/anonymize.ts`）
- [x] 「提供用データのプレビュー&手動ダウンロード」機能（設定 →「データ提供の準備」`/settings/data-provision`）
- [x] `userSecret` 生成と仮名ID導出（IndexedDB `meta` ストア、HMAC-SHA256）、バックアップJSON（version 3）への組み込み
- [ ] プライバシーポリシー整備・外部送信規律の通知文面ドラフト ← **Phase 1 着手前に必須**
- 依存: なし。**今すぐ着手可能**

### Phase 1 — パイロット（パートナー1社・手動提供）
**ゴール: 1社と組んで「提供 → レシート → 特典利用」を一周させる**
- [ ] パートナーレジストリ（partners.json + 署名 + アプリ側の取得/検証）
- [ ] 設定に「データ提供」セクション: パートナー一覧 → スコープ同意 → プレビュー → 送信
- [ ] レシート検証・保存、ポイント/クーポン表示画面
- [ ] パートナー向けAPI仕様書 + 受入テストツール（curl スクリプト or 簡易チェッカー）
- [ ] CSP方針の確定（connect-src の扱い）
- 依存: Phase 0、パートナー1社の合意、法務確認

### Phase 2 — 定常運用（複数パートナー・定期提供）
- [ ] 月次バッチ提供（月初にリマインド → ワンタップで先月分を送信）
- [ ] オファー取得（GET /offer）でキャンペーン表示
- [ ] 同意の期限管理・更新フロー、削除請求導線
- [ ] パートナー別ポイント台帳UI（累積・履歴・有効期限）
- 依存: Phase 1 の成功、パートナー3〜5社

### Phase 3 — エコシステム化
- [ ] パートナーのセルフサーブ申請フロー（PRテンプレート + 自動受入テストCI）
- [ ] 集計値のみモード（`stats.monthly`）の拡充 — 明細を出したくないユーザー向け
- [ ] iOS版との提供フォーマット共通化（`megroove-provision/v1` を共通仕様に）
- [ ] （オプション・要判断）中央リワードサーバーの検討 — 共通ポイント化はサーバーレス原則を破るため、やるなら別会社/別サービスとして分離

### 成立条件（KPIの目安）
- Phase 1 開始条件: MAU 数百人規模 or 熱量の高いコアユーザー100人（データの質が担保できる）
- パートナー価値提案: 「自宅抽出の実売後データ（どの豆がどう淹れられ、どう評価されたか）」はロースターにとって希少。カフェには「来店頻度×評価」のロイヤルティデータ。

## 5. アプリ側に必要な変更（実装観点まとめ)

| 変更 | 規模 |
|---|---|
| IndexedDB v3: `consents` / `receipts` ストア追加、`userSecret` 保存 | 小 |
| `src/provision/`: 匿名化・パッケージ生成・署名検証（WebCrypto: HMAC / Ed25519） | 中 |
| 設定画面「データ提供」タブ（同意管理・プレビュー・送信・台帳） | 中 |
| レジストリ fetch とキャッシュ（オフライン時は前回キャッシュ） | 小 |
| CSP調整・プライバシーポリシー画面 | 小 |

> Ed25519 は主要ブラウザの WebCrypto で利用可能（Safari 17+/Chrome 113+ 相当）。
> 非対応環境ではレシートを「未検証」フラグ付きで保存するフォールバックとする。
