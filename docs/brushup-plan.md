# ブラッシュアップ計画 — ストアTOP10級への引き上げ

コードベース全体（全ページ・主要コンポーネント）をレビューした上での、
「コーヒー好きがコーヒー人生を満足に楽しめるプロダクト」に到達するための改善計画。

CLAUDE.md の最優先方針（①記録の手軽さ → ②体験の心地よさ → ③再現性・分析 → ④共有は対象外）の順に
フェーズを構成する。**各フェーズ着手前に合意を取り、ブラウザで動作確認しながら進める。**

---

## 現状評価（2026-07 時点）

**強み**: 前回値プリフィル・その場追加・保存アニメーション・3表示モードなど中核体験は既に高水準。
コード品質も良好（型付け・後方互換処理・空状態の作り込み）。

**TOP10級との主な差分（コードレビューで確認した具体的事実）**:

1. **過去の一杯を記録できない** — `brewedAt` は常に `nowISO()`。「昨日の分を寝る前にまとめて記録」ができず、編集でも日時変更不可。記録アプリとして致命的な欠落。
2. **エラーが握りつぶされる** — DB書き込み失敗・写真読込失敗が `catch(() => {})` でユーザーに何も伝わらない（BrewPage:153 ほか多数）。保存できたつもりでデータが無い事故が起こり得る。
3. **データ消失リスクが放置** — `navigator.storage.persist()` 未使用。ブラウザ都合で IndexedDB が消え得るのに、バックアップ促進の仕組みもない（エクスポートは設定の奥）。
4. **抽出時間が「秒の数値入力」** — 淹れながら計れない。蒸らしタイマーはあるが総抽出のストップウォッチがない。
5. **豆の在庫が減らない** — 残量トラッキングがなく「あと何杯分あるか」「飲み切った豆のアーカイブ」ができない。こだわり派アプリの定番機能。
6. **分析が浅い** — レーダー＋トップ3のみ。時系列トレンド・豆ごとの比較・条件×評価の相関（=再現性の核心）がない。
7. **ライブラリに検索がない** — 豆名・カフェ名・メモの検索不可。月別グルーピングもなく、記録が増えるほど探せなくなる。
8. **アイコンが絵文字** — ☕🏪⚙📷≡⊞◉ 等が混在し、OS によって見た目が変わる。世界観（レコード×コーヒー）の統一を損なう最大の視覚要因。
9. **削除にアンドゥがない**、トースト等のフィードバック基盤がない。
10. **ホームに「今日」がない** — 本日の杯数・カフェイン残留量・連続記録日数など、毎日開く理由になる情報が出ていない。

## Phase A — 記録の信頼性と手軽さ（最優先）✅ 実装済み（2026-07-07）

> 「記録できて、絶対に失われない」というアプリの土台。
> 追加で実装: バックアップにカフェ記録が含まれていないバグを修正（backup version 2、旧ファイルも読込可）。

| # | 項目 | 内容 |
|---|---|---|
| A1 | 記録日時の編集 | 新規/編集どちらでも `brewedAt` / `visitedAt` を変更可能に（既定は現在時刻、タップで日時ピッカー）。「昨日の一杯」を記録できるようにする |
| A2 | トースト＋アンドゥ基盤 | 共通トーストコンポーネント。保存/削除/インポートの成否を必ず可視化。削除は5秒アンドゥ（実削除を遅延） |
| A3 | データ保全 | `navigator.storage.persist()` を初回に要求。設定に「最終エクスポートからN日」表示、30日超で控えめなリマインダー |
| A4 | 抽出ストップウォッチ | 蒸らしタイマーを拡張し「総抽出時間」を計測 → 停止で totalTimeSec に自動入力。mm:ss 表示に統一 |
| A5 | 豆の残量トラッキング | Bean に `initialAmountG` / `finishedAt` を追加。記録保存時に doseG を自動減算した残量を表示（「残り約◯杯」）。飲み切りでアーカイブ（ピッカーから消え、記録は保持） |

| # | 項目 | 内容 |
|---|---|---|
| C1 | 豆ごとの分析 | 豆詳細ページ: その豆の全記録・平均評価・ベストの一杯・レーダー・「この豆のベスト条件」（最高評価時の粉量/挽き目/湯温） |

## 実装しない/後回しにするもの

- 共有・SNS・コミュニティ（CLAUDE.md 方針④のとおりスコープ外）
- 多言語化（日本語ユーザー集中フェーズでは不要）
- サーバー/アカウント（サーバーレス原則）
- ドラッグ&ドロップ並び替え（既存の上下ボタン方式を維持）

## 進め方

1フェーズ = 1合意単位。各フェーズ内も項目ごとにコミットを分け、`npm run build` と実機ブラウザ確認を通す。
Phase A から順に着手し、A 完了時点で一度リリース（GitHub Pages デプロイ）するのを推奨。

### 着手前の確認事項

1. **A1 の日時ピッカー**: ネイティブ `<input type="datetime-local">` を世界観カラーで包む案を推奨（実装が軽く確実）。フルカスタムピッカーは工数大。
2. **A5 の残量**: 「購入時の量(g)」は任意入力とし、未入力の豆は従来どおり（残量表示なし）でよいか。
3. **B1 のアイコン**: 絵文字全廃でよいか。保存アニメーション等の演出系絵文字（紙吹雪）は維持想定。
4. **C3 の見せ方**: チャート（散布図系）まで作るか、まず「一言サマリ＋帯表示」の軽量版から始めるか（軽量版推奨）。
