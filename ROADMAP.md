# ROADMAP.md — Megroove 長期ロードマップ

**目標: コーヒーと音楽好きに、長く愛され続ける Megroove。**

## 0. この文書の使い方

- **CLAUDE.md は設計憲法（不変の判断基準）、本書は計画（可変）。** 矛盾したら CLAUDE.md が勝つ。
- 現在地・進行中の作業は **PROGRESS.md** を参照。本書は「これから」だけを扱う。
- **四半期ごと**に §4 のトレンド追従サイクルを回し、その結果で本書を更新する。
  完了した項目は消さず `✅ 完了（YYYY-MM）` を付けて残す（判断の履歴が次の判断の材料になる）。
- 新しい項目を足すときは、必ず「なぜ愛され続けることに繋がるか」を1行添える。
  理由が書けない項目はロードマップに載せない。

## 1. プロダクトの北極星（変わらない価値）

### 守り続けること

1. **「美味しかったあの一杯を、もう一度淹れられる」** — すべての機能はこのゴールに奉仕する。
2. **優先順位の憲法**（CLAUDE.md §2）: ①記録の手軽さ → ②体験の心地よさ → ③再現性・分析 → ④共有。
   迷ったら必ずこの順で判断し、手軽さを犠牲にする高機能化はしない。詳細機能は畳める前提で足す。
3. **データはユーザーのもの**:
   - **エクスポート（JSONバックアップ）は永久に無料。** データを人質にした課金はしない。
   - **ローカルのみで使い続ける自由を永続的に保証する。** 将来サーバー機能を導入しても、
     サーバーなしで中核体験（記録・分析・バックアップ）が完結する構成を維持する。
4. **レコード×コーヒーの落ち着いた世界観** — 上品なトーン、演出は軽く（保存演出は1.4秒枠）、
   `prefers-reduced-motion` 尊重。急かさない、騒がない。
5. **健康情報の規律**（CLAUDE.md §12） — 断定しない・推定と明記・出典・免責・外部ヘルスデータ非連携。

### アンチゴール（やらないことの明文化）

- **エンゲージメント稼ぎのプッシュ通知**（「今日はまだ記録していません！」の類）。
  開きたくなるのは中身の魅力によってであるべき。リマインドはユーザーが明示的に求めた場合のみ検討。
- **広告掲載。** 世界観と信頼を毀損する。収益は §3 の2軸のみ。
- **SNS的な承認数値**（いいね・フォロワー・公開ランキング）。共有は方針④のとおり当面スコープ外で、
  導入する場合も「比較」ではなく「見せたい一杯を見せる」形に限る。
- **過剰なゲーミフィケーション**（毎日ログインボーナス等）。節目演出（100杯目など)のような
  「静かな祝福」までに留める。
- **ダークパターン**（解約妨害・誇張バッジ・不安を煽る文言）。健康表現の規律と同根。

## 3. 収益化ロードマップ

2軸: **フリーミアム（Pro サブスク）** と **データ提供リワード**。広告はやらない（§1 アンチゴール）。

### 無料で守り続ける中核（= 愛される条件。将来も有料化しない）

- 記録（ブリュー/カフェ）・前回値プリフィル・クイック記録・保存演出
- 基本分析（レーダー・ランキング・累計統計）・カフェイン管理
- **エクスポート/インポート（バックアップ）— 永久無料**
- ストック管理・産地パスポート・記録画面カスタマイズ

### Pro（サブスク）候補 — 「深さ」と「装い」を有料に、「記録」は無料のまま

- 高度分析: 飲み頃分析の詳細版・条件×評価の相関エクスプローラ・長期トレンド
- Year in Coffee の高解像度書き出し・過去年ぶん生成
- レコード盤スキン/テーマ（ラベルデザイン・配色バリエーション）
- （サーバー導入後）複数端末同期

### 段階定義

| Stage | 状態 | 収益 | サーバー |
|---|---|---|---|
| **0（現在）** | 完全無料・ローカル完結 | なし | 不要 |
| **1** | 応援課金 or Pro 買い切り/サブスク（Stripe Payment Links ＋ ライセンスキー等、サーバーレスで可能な範囲） | Pro | 不要〜最小 |
| **2** | データ提供リワード開始 | Pro＋提供パートナー | 最小構成（提供受付のみ） |
| **3** | 複数端末同期サブスク | Pro（同期含む） | 本格 |

### ローカル完結からの移行条件（すべて満たすまで移行しない）

1. 明確な必要性: データ提供パートナーの確定（Stage 2）**または** 同期需要の顕在化（Stage 3）
2. プライバシーポリシーの改定と、アプリ内での**明示的な事前告知・同意**
3. **ローカルのみモードの永続保証**: サーバー機能はすべてオプトイン。拒否しても中核体験は損なわれない
4. 送信データの最小化と可視化（Phase D の匿名化エンジン・プレビュー方式を踏襲）

## 4. トレンド追従の運用の型（四半期サイクル）

### 手順書（1サイクル = 半日想定）

1. **調査**: 下のプロンプトテンプレート4本を Claude Code に順に投げる（WebSearch 可能な環境で）
2. **候補化**: 調査結果から機能候補を最大10件、各1行の「何を・なぜ」で列挙させる
3. **北極星フィルタ**: 各候補を次のチェックリストで採点し、通過したものだけ残す
   - [ ] 記録の手軽さを損なわないか（増える入力は任意・畳めるか）
   - [ ] ローカル完結で実装できるか（できないなら §3 の移行条件を満たすまで保留）
   - [ ] 健康情報の規律（CLAUDE.md §12）に抵触しないか
   - [ ] 演出なら軽量か（transform/opacity 中心・reduced-motion 対応・体験時間を延ばさない）
   - [ ] 世界観（レコード×コーヒー・落ち着いた上品さ）に合うか
   - [ ] アンチゴール（§1）に触れないか
4. **選定と実装**: 1〜3件だけ選び、従来どおり「計画→合意→実装→確認→コミット」
5. **更新**: 本書の該当フェーズに反映（採用も見送りも理由つきで記録）

## 5. 技術的な持続性

### 依存パッケージの更新方針

- **四半期ごと**（§4 のサイクルと同時）に `npm outdated` を確認
- patch/minor: 原則追従。`npm run build` ＋ 主要動線の手動確認（記録→保存→ライブラリ→分析→エクスポート/インポート）をセットで
- major（React/Vite/Tailwind/idb/vite-plugin-pwa): リリースノートを読み、**1メジャー遅れまで許容**。
  急がない — 動いているアプリの安定が最優先。移行は専用コミットに分離
- セキュリティアドバイザリのみ即時対応

### IndexedDB マイグレーション方針

- **原則: 任意フィールドの追加で済ませ、DBバージョンは上げない**（Phase A〜E はすべてこの方式で
  スキーマ変更ゼロを維持。scene/drinkStyle/decaf 等が実例）
- DB version++ が必要なのは「ストア追加・インデックス追加/変更」のみ。その際は:
  - `upgrade` コールバックは古い version からの段階的な移行をすべて処理する（switch fallthrough 形式）
  - データ変換は**べき等**に書く（2回実行しても壊れない）
  - 変更前に必ずエクスポートでの復旧経路を確認する
- 読み取り側は常に「フィールドが無い古いデータ」を許容する（`?.` と既定値。既存コードの流儀を踏襲）

### バックアップ（JSON）互換性の維持ルール

- エクスポートに `version` フィールド必須（現行 version 2）
- **インポートは過去すべての version を永久にサポートする**（version 1 の読込対応が実績）
- フィールドの追加だけなら version は据え置き。**意味が変わる変更**（単位変更・必須化・構造変更）のみ version++
- `userSecret`（データ提供の仮名ID用）をバックアップに含める仕様を維持（機種変更対応）
- インポート実装を変更したら、必ず「旧バージョンのサンプルJSON」での読込確認を行う

### セキュリティ・その他

- 本番 CSP（`style-src 'self'` 等）は維持。dev のみ `vite.config.ts` のプラグインで緩和（経緯は PROGRESS.md）
- 写真は base64 800px リサイズの現行方式を維持（容量とのバランス。変更するなら移行とセットで）
- テスト: 現状は手動確認ベース。導入するなら **`stats.ts`・`helpers.ts` の純関数から vitest** が費用対効果最良
  （分析ロジックはリグレッションが起きやすく、UIなしで検証できる）

## 6. 公開後のグロース戦略

公開後にどう育てるかの計画。**順番を守って育てる**のが根幹で、ユーザー不在で収益化を急がず、
未成熟なうちに宣伝に走らない。**最重要指標はユーザー数ではなく継続率**。
Megroove は記録の蓄積で価値が出る設計なので、数を追わなくても継続率で戦える。
発信やグロースにも、§1 の北極星とアンチゴール、CLAUDE.md §2 の優先順位（手軽さ第一）をそのまま適用する。

> **継続率の運用定義**: *ある月に1回以上記録した人のうち、翌月も1回以上記録した割合。*
> この翌月継続率を見て「広げる」か「磨く」かを判断する。

### 6.1 基本原則

- **順番を守る** — ユーザー不在で収益化を急がない。未成熟なまま宣伝に走らない。段階を飛ばさない。
- **継続率が最重要指標** — 上の運用定義のとおり。フォロワー数・DL数・MAU といった規模の数字を目標にしない。
- **アンチゴールを発信にも適用**（§1）— 煽らない・盛らない・広告しない・急かすプッシュ通知をしない。
  発信は「見せたい一杯を見せる」トーンで、静かに続ける。

### 6.2 段階（4フェーズ）

各段階を「やること / 目標 / 次段階へ進む条件」で持つ。**条件を満たすまで次に進まない**。

#### 第1段階　自分が最初の最高のユーザーになる（〜1ヶ月）

- **やること**: 宣伝より先に、作者自身が毎日記録をつける。不具合・使いにくさを自分で先に潰す。
  実データで映えるスクリーンショットを撮り溜める（レーダーチャート／ターンテーブル計測／
  産地パスポート／飲み頃分析など、後の発信素材にする）。
- **目標**: 作者自身が「毎日開きたい」状態になっていること。発信素材のストックができていること。
- **次段階へ進む条件**: 主要動線（記録→保存→ライブラリ→分析→バックアップ）に致命的な不具合がない・
  作者自身が継続できている・発信素材が数点そろった。

#### 第2段階　最初の10人 — 作品として差し出す（〜3ヶ月）

- **やること**: 広告ではなく、コーヒー好きのいる場所（X のコーヒー界隈・コミュニティ）へ
  「作品＋開発の物語」として差し出す。物語こそ共感の武器 —
  *Apple Developer 登録に阻まれ Web で公開した経緯／レコード×コーヒーの世界観／手軽さへのこだわり*。
  あわせて身近なコーヒー好き（行きつけのバリスタ・友人）に直接使ってもらう。
- **目標**: 数を集めることではなく、「勧められた他人が使い続けるか」を確認すること。
- **次段階へ進む条件**: 作者以外で、翌月も使っている人が複数人いる。

#### 第3段階　100人へ — 発信の習慣化と、声で磨くサイクル（〜6ヶ月）

- **やること**: 継続発信を核にする。宣伝ではなく「アプリで見えた自分のコーヒー体験」を週1〜2回発信する
  （例: 今月のゴールデンレシピ／「この豆は焙煎後10日がピーク」＝飲み頃分析の画面つき）。
  フィードバック窓口（Google フォーム等）を用意し、集まった声を §4 の四半期トレンド追従サイクルの
  判断材料に接続する。
- **目標**: 継続ユーザーの母数を増やしつつ、「声で磨く」ループを回す。**この段階まで収益ゼロが正常**。
  焦って課金を入れない。
- **次段階へ進む条件**: 継続ユーザーが安定して増加している・声で磨くべき点が一巡した。

#### 第4段階　収益化を試す — §3 Stage 1 から小さく（6ヶ月〜）

- **やること**: いきなりサブスクにしない。まず**応援課金**から（Buy Me a Coffee 型。コーヒーアプリと相性が良い）。
  応援が集まってきたら **Pro サブスク**へ（§3「Pro（サブスク）候補」の高度分析・Year in Coffee 高解像度書き出し・
  レコード盤スキン等）。**無料の中核は死守**する（記録・前回値プリフィル・保存演出・基本分析・バックアップ）。
  これは §3 の **Stage 1** に対応する。
- **目標**: 中核無料を守ったまま、小さく収益化を検証する。
- **次段階へ進む条件**: **データ提供リワード（§3 Stage 2）はさらに先**。ユーザー規模と信頼が前提で、
  §3「ローカル完結からの移行条件」を**すべて**満たすまで着手しない。着手時は**法務専門家への相談を必須**とする。

### 6.3 貫く3つの規律（地味だが重要）

- **継続率を最重要指標に** — 翌月継続率（本節冒頭の運用定義）を見て、「広げる」フェーズに進むか「磨く」に留まるかを判断する。
- **kill criteria を持つ** — 撤退・見直しの線を先に決めておく。例: *6ヶ月発信して継続ユーザーが10人未満なら戦略を見直す*。
  惰性で続けない判断材料にする。
- **作者が楽しみ続けられる設計** — 個人開発最大のリスクは作者の疲弊。「コーヒーと音楽が好きだから続く」形を保つ。
  グロースのために体験の楽しさや自分の熱を犠牲にしない。
